# Origami Smart PM — documentation

This folder is the **source-of-truth documentation** for the Origami Smart Project Management app. It is written so you can **upload or @-reference files in AI chat** (ChatGPT, Cursor, and similar tools) and **edit the Markdown directly** in those sessions or in the editor.

---

## Canonical file order (read this order)

Use this sequence whenever you onboard someone—or when you paste docs into an AI context window. **Order matters:** the PRD defines *what* you are building; the Build Guide defines *how* to implement it in Origami.

| Step | File | Role |
|------|------|------|
| **1** | **README.md** (this file) | How the doc set is organized, how to use it with AI, and where everything lives. |
| **2** | **[PRD.md](PRD.md)** | **PRD** — Product Requirements Document: vision, scope, entities, pages, workflows, QA, budget logic. |
| **3** | **[BUILD_GUIDE.md](BUILD_GUIDE.md)** | **Build Guide** — step-by-step Origami setup: users, entities, permissions, pages, workflows, Gantt, QA board, email. |

**Note on naming:** If you see “PRD and PDR” elsewhere, the second artifact here is the **Build Guide** (`BUILD_GUIDE.md`), not a separate “PDR” file. The pair you maintain is **PRD + Build Guide**.

## Merge into one file (export for other platforms / AI)

The **source of truth stays split** (`README.md`, `PRD.md`, `BUILD_GUIDE.md`). When a tool only accepts **one upload** or you want a single `.md` to email or archive, build a **throwaway combined file** from the repo root (`Origami/`).

### Option A — PRD + Build Guide only (best for “full spec” uploads)

Most of the time you only need the two substantive documents. Internal anchors (`#prd-1`, `#build-1`, etc.) stay valid inside the merged file. Cross-file links such as `[PRD](PRD.md)` will not resolve in a single file; models usually still understand the context.

```bash
cd /path/to/Origami
printf '%s\n' "# Origami Smart PM — PRD + Build Guide (merged export)" "" "_Merged from PRD.md and BUILD_GUIDE.md. Do not edit this file; edit the sources and regenerate._" "" "---" "" > ORIGAMI_SMART_PM_MERGED.md
cat PRD.md >> ORIGAMI_SMART_PM_MERGED.md
printf '\n\n---\n\n' >> ORIGAMI_SMART_PM_MERGED.md
cat BUILD_GUIDE.md >> ORIGAMI_SMART_PM_MERGED.md
```

### Option B — README + PRD + Build Guide (full bundle)

Use this when the consumer should also get **how this doc set is meant to be used**.

```bash
cd /path/to/Origami
printf '%s\n' "# Origami Smart PM — full documentation (merged export)" "" "_Merged from README.md, PRD.md, and BUILD_GUIDE.md. Do not edit this file; edit the sources and regenerate._" "" "---" "" > ORIGAMI_SMART_PM_FULL_EXPORT.md
cat README.md >> ORIGAMI_SMART_PM_FULL_EXPORT.md
printf '\n\n---\n\n' >> ORIGAMI_SMART_PM_FULL_EXPORT.md
cat PRD.md >> ORIGAMI_SMART_PM_FULL_EXPORT.md
printf '\n\n---\n\n' >> ORIGAMI_SMART_PM_FULL_EXPORT.md
cat BUILD_GUIDE.md >> ORIGAMI_SMART_PM_FULL_EXPORT.md
```

### After merging

- Add `ORIGAMI_SMART_PM_MERGED.md` / `ORIGAMI_SMART_PM_FULL_EXPORT.md` to **`.gitignore`** if you do not want exports committed, or commit them if your team wants a downloadable artifact.
- **Regenerate** the merged file whenever you change `PRD.md` or `BUILD_GUIDE.md`.
- For **ChatGPT / other AI**: upload the generated `.md` or paste its contents; for **Cursor**, you can also @-mention the three source files instead of merging.

### Windows (PowerShell)

From the `Origami` folder, PRD + Build Guide only:

```powershell
@"
# Origami Smart PM — PRD + Build Guide (merged export)

_Merged from PRD.md and BUILD_GUIDE.md. Do not edit this file; edit the sources and regenerate._

---

"@ | Out-File -Encoding utf8 ORIGAMI_SMART_PM_MERGED.md
Get-Content PRD.md | Add-Content -Encoding utf8 ORIGAMI_SMART_PM_MERGED.md
"`n`n---`n`n" | Add-Content -Encoding utf8 ORIGAMI_SMART_PM_MERGED.md
Get-Content BUILD_GUIDE.md | Add-Content -Encoding utf8 ORIGAMI_SMART_PM_MERGED.md
```

## Working with AI (ChatGPT, Cursor, etc.)

1. **Attach or reference the whole set in order:** start with this README, then `PRD.md`, then `BUILD_GUIDE.md` when the model needs implementation detail.
2. **Ask for edits in place:** e.g. “Update section X in PRD.md to add field Y” — then apply the diff in Cursor or paste the updated Markdown back into ChatGPT projects/files if you use them.
3. **Cross-file links** in these Markdown files use relative paths (`PRD.md`, `BUILD_GUIDE.md`). They resolve in GitHub, VS Code / Cursor preview, and most static doc hosts when the folder is published as a repo or site root.
